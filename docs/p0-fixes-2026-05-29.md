## P0-1 — Dark-mode semantic colors + tab-tint contrast

**What changed** (`DesignSystem/BeautyDesignSystem.swift`, `App/RootView.swift`):

- `success` and `danger` converted from static `Color(...)` to `dynamic(light:dark:)`. Previously they used a single light-tuned value that fell below 4.5:1 on the dark `card` surface; each now has a lightened dark variant.
- `orange` converted to dynamic (was static); dark variant lifted for the dark surface.
- `warmGray` retuned: light value darkened (was failing as body/secondary text), dark value lightened.
- New `limeTint` token for the **selected tab** (`MainTabView.tint`). The raw brand `lime` as a tab tint measured ~1.3:1 against the bar — invisible as a selection state. `limeTint` is a deep chartreuse on light / bright lime on dark. `lime` itself is unchanged and still used for filled buttons (dark ink on bright lime = 12.66:1).
- `limeInk` left static intentionally — it only ever sits on the bright `lime` fill.

**Recomputed WCAG 2.1 ratios** (sRGB, against the surface each color actually renders on):

| Token | Mode / surface | Ratio | Threshold | Pass |
|---|---|---|---|---|
| warmGray | light / card | 5.10:1 | 4.5 (text) | ✅ |
| warmGray | dark / card | 5.80:1 | 4.5 | ✅ |
| danger | light / card | 5.96:1 | 4.5 | ✅ |
| danger | dark / card | 6.67:1 | 4.5 | ✅ |
| success | light / card | 6.22:1 | 4.5 | ✅ |
| success | dark / card | 8.58:1 | 4.5 | ✅ |
| limeTint | light / bar (non-text) | 4.92:1 | 3.0 | ✅ |
| limeTint | dark / bar (non-text) | 10.97:1 | 3.0 | ✅ |
| orange | light / card | 3.69:1 | — | decorative only* |

*`orange` appears only as a 0.14-opacity accent pill, not as text or a control state — below 4.5:1 is acceptable for non-text decoration. Flagged, not blocking.

## P0-2 — Touch targets ≥ 44 pt (WCAG 2.5.5)

Five genuinely-interactive controls under 44 pt were padded to a 44×44 hit area while keeping their visual size (added `.frame(minHeight/​width: 44)` + `.contentShape(Rectangle())` *after* the visual background):

| Control | Visual | Hit area |
|---|---|---|
| Cart quantity +/− (`CartView`) | 32×32 circle | 44×44 |
| Advisor mini-card action (`AdvisorView`) | h28 capsule | h44 |
| Advisor tray action (`AdvisorView`) | h38 capsule | h44 |
| Product shelf buttons (`ProductDetailView`) | h38 capsule | h44 |
| Photo-scan close ✕ (`PhotoScanView`) | circle | 44×44 |

Cart +/− also gained `accessibilityLabel` ("Добавить ещё один" / "Убрать один") since the glyph-only buttons had no spoken label. Decorative/label-only elements (MatchBadge, index/icon badges, full-row chevrons, trust seal) were deliberately left — they are not tap targets.

## P0-3 — Copy leaks

- «Checkout Золотого Яблока…» → «Оформление заказа пока не подключено» / «Список к покупке…» (`CartView`, 3 strings).
- «demo / demo-каталог / demo-список» → neutral «Каталог / список» wording across `AppState`, `RecommendationsView`, `BeautyModels`, `HomeView`, `ProfileView`, `ProductDetailView`.
- `support@example.com` → `support@lumabeautyid.app` (`SettingsView`).
- Hard-coded «Версия 1.0 RC» → `versionLabel`, read live from `Info.plist` (`CFBundleShortVersionString`/`CFBundleVersion`) so the displayed version always tracks the build config.

Residual `example.com` matches are non-user-facing and intentional: the dev-login fixture email (`development@example.com`, debug builds only), an env-validation guard (`api.example.com`), and test fixtures.

## P0-4 — Unify the three card abstractions

`beautyCard()`, `HomeSurfaceCard`, and `ProfileSurfaceCard` were three near-identical surfaces. Unified into one `SurfaceCard<Content>` view (params: `tint`, `borderOpacity`, `cornerRadius`) in the design system:

- `beautyCard()` is now a thin convenience wrapper that routes through `SurfaceCard`, so existing call sites keep working.
- `HomeSurfaceCard` (7 call sites) and `ProfileSurfaceCard` (8 call sites) deleted; calls replaced with `SurfaceCard`, preserving the `featuredCard` tint / custom border at each site.
- Net: one source of truth for card styling, identical rendered output.
